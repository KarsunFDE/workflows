Finishing the Galent 7-Step Workflow

Three groups · minimal overlap · target: cover every unfinished step by tomorrow

Our workflow copies Galent's 7-step app-modernization process. Steps 1 and 2 are done — we can already ingest a project and analyze it. The remaining steps (3 through 7) are either partly built or not built at all. This sprint finishes them.

Each gap is assigned so that each group works in a different file — no two people edit the same file, so there are no merge conflicts on GitHub. The work extends the existing command chain: /fde-personas → /fde-analyze → /fde-plan, plus a brand-new /fde-implement command for the build steps.

Added capability — cloud architecture. The workflow must also map out how the modernized app runs in the cloud: packaged into Docker containers running on Amazon EKS (Elastic Kubernetes Service). That work splits along the same chain and the same owned files — inventory (Group 3) → design (Group 2) → manifests (Group 1) — so nothing new collides.

The two cross-group handoffs

Plain-text handoffs (so no one blocks anyone)

Group 3 → Group 2: Group 3 stamps finding IDs (F-001…); Group 2's review gate checks each story points back to one. Agree the ID format at the start of the day.

Group 2 → Group 1: Group 1's new command reads Group 2's plan (its stories/subtasks) as input. Group 1 can build against a small sample plan until Group 2's lands, so the two never wait on each other.

Cloud architecture (☁) chain: Group 3 inventories the containers/infra under ## Deployment & Container Inventory → Group 2 designs the Amazon EKS deployment from it → Group 1 emits the Docker + EKS manifests. Each stage lives in the file that group already owns, so the firewall holds — no new shared file.

Ground rules — why this avoids merge conflicts

  • One file per group. G3 → fde-analyze.js, G2 → fde-plan.js, G1 → new fde-implement.js. No file touched twice.
  • Test on your own project copy. Each team runs against its own clone.
  • Profiles are settled. Grants is done; no persona regeneration is part of this sprint.
  • Don't edit the shared design docs (combined_workflow.md, current-plan.md) during the sprint — the lead folds in status at the end.
  • Branch off the current main. It already has the latest fixes merged.
  • One pull request per group. Disjoint files → merge in any order.
